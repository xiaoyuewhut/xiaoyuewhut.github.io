引言#

UDS 本质上是一套让外部设备和 ECU 有组织地对话的标准语言。

这里的外部设备，可以是产线刷写设备、售后诊断仪、研发调试工具，也可以是台架上的自动化测试脚本。这里的 ECU，也可以是发动机控制器、制动控制器、域控制器、车身控制器、摄像头、雷达，甚至网关。

如果把它类比成医生问诊，会更容易理解：

  • “你现在是什么状态？” 对应读取数据、读取故障码。
  • “把错误清掉。” 对应清故障码。
  • “进入维修模式。” 对应切换诊断会话。
  • “证明你有权限。” 对应安全访问。
  • “开始做手术。” 对应例程控制、刷写、复位、标定。

UDS 的常见应用#

UDS 在工程里最常见的应用大致可以分成五类：

  • 健康检查：例如读取 DTC 故障码、读取冻结帧、读取传感器状态、确认某个 ECU 是否在线、软件版本是否正确、标定号是否正确。
  • 调试与验证：研发联调时经常要读内部变量、触发某个例程、强制执行器动作、临时关闭某个策略、检查某个条件是否满足。
  • 生产制造：产线要给 ECU 写入软件、写入 VIN、写入配置参数、写入密钥、做 EOL 标定、运行自检。
  • 售后维修：售后诊断仪会读取故障、清除故障、做部件更换后的匹配，并执行一些服务功能，例如转向角学习、制动系统排气、摄像头标定。
  • 软件升级：这是 UDS 最常见、也最重要的应用之一。很多 ECU 的刷写流程，本质上就是一套固定顺序的 UDS 服务调用。

UDS 报文的基本结构#

无论诊断场景多复杂，UDS 的基本报文思路其实很统一。

一条请求报文通常包含：

  • SID：Service Identifier，服务标识。
  • Sub-function：子功能，不是每个服务都有。
  • Parameter：参数区。

例如发送一条 0x22 F1 90，其含义通常是：

  • 0x22：读取数据标识 ReadDataByIdentifier
  • F1 90：要读的数据 DID 是 0xF190

在很多项目里，0xF190 表示 VIN。

正响应#

如果请求服务号是 0x22，成功响应通常会是 0x62；如果请求服务号是 0x10，成功响应通常会是 0x50。

也就是说：

正响应 SID=请求 SID+0x40\text{正响应 SID} = \text{请求 SID} + 0\mathrm{x}40正响应 SID=请求 SID+0x40

所以抓包时看到 62 F1 90 ...，经验上就能很快反应过来，这是 0x22 读 DID 的成功响应。

会话机制#

UDS 最重要的工程思想之一，是不同权限、不同风险的功能，必须在不同会话里开放。

一个 ECU 不可能在默认状态下允许任何人随意：

  • 清故障
  • 改配置
  • 控制执行器
  • 刷程序

因此，UDS 引入了 Diagnostic Session，也就是诊断会话的概念。

最常见的几个会话是：

  • Default Session：默认会话，ECU 正常工作时的基础诊断状态。
  • Extended Diagnostic Session：扩展诊断会话，开放更多调试和诊断能力。
  • Programming Session：编程会话，主要用于软件刷写。

对应的服务是 0x10，即 DiagnosticSessionControl。

例如：

  • 请求 10 01：进入默认会话
  • 请求 10 03：进入扩展会话
  • 请求 10 02：进入编程会话

如果成功，可能返回：

  • 50 03 ...：成功进入扩展会话

这里的后续字节通常还会带上 P2 / P2* 时间参数，用于告诉测试设备超时策略。

0x27 安全访问#

只靠会话还不够，因为不能允许任何设备都能切进高权限会话，然后随便刷车。

所以 UDS 里还有一个特别关键的服务：0x27，也就是 SecurityAccess。

它的典型工作方式是 Seed / Key 机制，流程通常如下：

  1. 测试设备请求一个 seed
  2. ECU 返回 seed
  3. 测试设备根据某种算法把 seed 算成 key
  4. ECU 校验 key
  5. 校验通过后，开放对应权限

一个很常见的交互序列是：

  • 27 01：请求 level 1 的 seed
  • ECU 返回 67 01 xx xx xx xx
  • 测试设备计算后发送 27 02 yy yy yy yy
  • ECU 若认可，返回 67 02

常见误区#

很多人会以为进了扩展会话就一定能刷写，其实通常不是这样。真实项目里往往是：

  1. 先进入某个会话
  2. 再做安全访问
  3. 再检查若干前置条件
  4. 然后才允许真正执行某个高权限服务

所以诊断失败时，不能只看服务号，还要看：

  • 当前会话对不对
  • 安全等级够不够
  • 前置条件是否满足
  • ECU 是否被网关或系统状态限制

0x22 与 0x2E：读写 DID#

UDS 里一个非常重要的概念是 DID，全称 Data Identifier。

可以把 DID 理解成一个被标准化或项目约定好的数据项编号。测试设备不需要知道 ECU 内部变量放在什么地址，而是通过 DID 来访问。

例如：

  • 0xF190 常用于 VIN
  • 0xF187、0xF189 这类 DID 常被用于软件版本、硬件号、标定号等信息

具体定义会因标准和 OEM 约定不同而变化。

写 DID：0x2E#

如果某些 DID 被允许写入，就会用 WriteDataByIdentifier，服务号 0x2E。

但不要误会，不是所有 DID 都能写。多数关键 DID：

  • 要求特定会话
  • 要求安全解锁
  • 要求满足前置条件
  • 有的数据甚至出厂后就不允许再写

所以 0x2E 在工程上比 0x22 敏感得多。

0x19 与 0x14：故障码相关服务#

DTC 是什么#

DTC 是 Diagnostic Trouble Code，即诊断故障码。它不是一句自然语言，而是 ECU 内部对某类故障状态的编码表示。

例如某个传感器开路、通信超时、内部校验失败、电压异常，都可能映射成某个 DTC。

读 DTC：0x19#

UDS 中读取故障信息最核心的服务是 ReadDTCInformation，服务号 0x19。

这个服务很大，因为它不只是把故障码列出来那么简单，还会涉及：

  • DTC 本身
  • DTC 状态位
  • 快照信息 snapshot
  • 扩展数据 extended data
  • 故障计数
  • 故障发生条件

也就是说，DTC 不是一个孤零零的编号，它经常还带着一组上下文信息。

清 DTC：0x14#

清故障一般使用 ClearDiagnosticInformation，服务号 0x14。

听起来很简单，但工程上它并不是按一下按钮就世界清净。因为清故障是否成功，取决于很多因素：

  • 故障当前是否仍然存在
  • ECU 是否允许在当前状态下清除
  • 网关或系统策略是否限制
  • 某些故障是否属于不可直接清除类型

所以你会看到一种很常见的现象：工具提示清除了，但刷新后故障又出现了。

这通常不是工具有问题，而是根因没有消失。监测条件再次满足后，ECU 又把故障重新置位了。

0x31：例程控制#

UDS 里还有一个很常用的服务：0x31，RoutineControl。

可以把它理解成：不是直接读一个数据，也不是直接写一个数据，而是请 ECU 执行一段已经内置好的诊断动作。

这类动作可能包括：

  • 擦除 Flash 某个区域
  • 校验程序完整性
  • 标定某个传感器
  • 执行某个自检
  • 切换某种工厂模式

它通常会带一个 RID，也就是 Routine Identifier，用于标识具体执行哪个例程。

常见子功能包括：

  • startRoutine
  • stopRoutine
  • requestRoutineResults

这很好理解，因为有些例程不是瞬间完成的，例如自检、擦除、标定、校验，都可能需要时间。所以很多复杂动作在诊断里不会设计成“一条命令立刻完成”，而是设计成：

  1. 先启动
  2. ECU 处理中
  3. 必要时查询结果

这比简单粗暴地设计成单条命令更符合工程实际。

执行器控制与通信控制#

当你做联调、售后服务功能或者台架测试时，经常会遇到 ECU 不只要把信息告诉你，还要真的做一个动作。

0x2F：InputOutputControlByIdentifier#

这个服务常用于对某个输入输出对象做控制，例如临时强制某个输出、让某个执行器按指定方式动作。

一个直观例子是：如果要验证风扇、继电器、指示灯、电机、阀门等动作链路是否正常，工程上可能就会设计成某个 0x2F 控制项。

0x28：CommunicationControl#

这个服务用于控制 ECU 的通信行为，在刷写或特殊调试场景里比较常见，例如临时关闭某类正常通信，避免干扰诊断过程。

0x85：ControlDTCSetting#

这个服务用于控制 DTC 的设置行为。在某些开发、生产或刷写场景中，为了避免过程中的临时异常产生大量无意义故障码，可能会临时关闭 DTC 记录。

但这类能力都比较敏感，因为它们会影响 ECU 正常监控或对外表现，所以通常不会在低权限下开放。

0x3E：保持会话#

抓包时经常会看到一条看起来很无聊的报文：3E 00。这就是 TesterPresent，服务号 0x3E。

它的作用是告诉 ECU：测试设备还在线，不要退出诊断状态。

之所以需要它，是因为很多 ECU 进入扩展会话或编程会话后，并不会永久停留在那里。如果一段时间收不到诊断设备的活动，它会自动退回默认状态，以降低风险。

所以测试设备通常会周期性发送 3E 00 来保活。

UDS 里几个常见的时间概念是：

  • P2：正常情况下 ECU 最晚多久给出响应
  • P2*：如果 ECU 需要更久处理，扩展后的等待时间
  • S3：诊断会话保持时间，长时间没动静就可能退会话

这些都是写自动化脚本时需要特别关注的。

负响应码 NRC#

新手最该优先学会读的，往往不是服务表，而是负响应码。

因为很多现场问题，最后并不是 ECU 完全不懂你的请求，而是它懂，但它此刻不允许你这么做。

NRC常见含义你可以怎样理解
0x10General RejectECU 拒绝了，但没给更具体原因
0x11Service Not Supported这个服务 ECU 不支持
0x12Sub-function Not Supported服务支持，但这个子功能不支持
0x13Incorrect Message Length or Invalid Format报文长度或格式不对
0x22Conditions Not Correct条件不满足，比如电压、车速、档位、状态不对
0x31Request Out Of Range参数越界、DID / RID 不存在，或当前范围不允许
0x33Security Access Denied没权限，或者安全状态不对
0x35Invalid Key你算出来的 key 不对
0x36Exceed Number Of Attempts安全访问失败次数过多
0x37Required Time Delay Not Expired还在等待期，不能立刻再试
0x78Response PendingECU 正在处理，让你再等等

特别强调 0x78 的意义。它不是失败，而更像是 ECU 在说：

我已经收到，也会处理，但这件事比较慢，你先别超时。

这在刷写、擦除、复杂例程、自检场景中都很常见。如果测试脚本不能正确处理 0x78，就很容易误判为失败。

UDS over CAN 的帧类型#

只要开始看 CAN 抓包，就一定会碰到一个现实问题：很多 UDS 数据根本塞不进单帧 CAN。这个时候 ISO-TP 就开始工作了。

四类典型帧#

ISO-TP 最核心的几类帧是：

  • Single Frame：单帧，数据较短时一次发完
  • First Frame：首帧，告诉对方这是一条长消息，总长度是多少
  • Consecutive Frame：连续帧，后续分段数据
  • Flow Control：流控帧，接收方告诉发送方怎么继续发

为什么需要 Flow Control#

因为长消息发送时，接收方未必来得及一直收。所以接收方可以通过流控帧告诉发送方：

  • 可以继续发
  • 一次最多连续发多少帧
  • 相邻帧之间至少间隔多久

这就是你在诊断抓包里常会看到的：

  • BS：Block Size
  • STmin：最小帧间隔时间

例子#

假设 ECU 要返回一个 VIN，长度超过单帧上限，那么过程可能像这样：

  1. ECU 先发首帧，告诉你总长度 20 字节
  2. 测试设备回流控帧，告诉 ECU 可以继续发
  3. ECU 再连续发送若干连续帧
  4. 测试设备在上层把这些帧重新拼成一条完整 UDS 响应

所以当你在 CAN 日志里看到一堆分散的帧时，不要急着逐帧按 UDS 去解读。很多时候首先要做的，是先在脑子里把 ISO-TP 重组出来。

物理寻址和功能寻址#

UDS 诊断请求并不总是一对一发送。

Physical Addressing：物理寻址#

这是最常见的方式，也就是请求明确发给某一个 ECU。例如要读取某个具体控制器的版本号、故障码或内部数据，通常都会用物理寻址。

它的特点是：

  • 目标明确
  • 一般由那个 ECU 单独响应
  • 适合大多数具体操作

Functional Addressing：功能寻址#

这更像一类按功能广播的请求。请求发出去后，网络中所有支持该功能且愿意响应的 ECU，都可能按规则处理。

它常用于：

  • 某些统一发现类请求
  • 某些法规或通用诊断场景

但功能寻址并不适合所有服务，尤其不适合那些会改变 ECU 状态的敏感操作。

所以在工程实践里，一个很重要的意识是：不是所有 UDS 服务都能随便配任何寻址方式。

软件刷写#

UDS 刷写软件详解 如果做域控、中央计算平台、网关、传感器 ECU，软件下载和升级几乎一定会碰到 UDS。

一个典型的 UDS 刷写流程通常是：

  1. 进入编程会话 0x10
  2. 做安全访问 0x27
  3. 可能执行某些前置例程 0x31
  4. 发起下载请求 0x34，告诉 ECU 要下载的数据格式和地址范围
  5. 使用 0x36 连续发送数据块
  6. 用 0x37 告诉 ECU 传输结束
  7. 可能执行完整性校验或依赖检查例程 0x31
  8. ECU 复位 0x11

这里最关键的理解不是死记服务号，而是搞清楚为什么它要分这么多步。

原因在于刷写本身就是一个高风险动作：

  • 必须确保进入正确模式
  • 必须确保有权限
  • 必须告诉 ECU 要写多少、写到哪里
  • 必须分块传输
  • 必须让 ECU 验证数据是否完整
  • 最后还得让 ECU 重新启动并进入新软件

所以它天然就不可能是“一条命令把包扔过去”那么简单。

为什么刷写阶段经常看到 0x78#

因为很多动作确实需要时间，例如：

  • 擦除 Flash
  • 写入大块数据
  • 做 CRC 校验
  • 做程序完整性检查

这时 ECU 很可能先回复 0x7F xx 78，表示它还在忙。测试设备如果能正确处理这个节奏，刷写才会稳定。
