需求分解是系统工程师最重要的工作之一。它把一句功能目标拆成项目里不同角色能执行的内容：架构工程师据此分配系统责任，软件工程师据此写逻辑，硬件和底盘工程师据此确认能力边界，测试工程师据此设计测试用例，项目管理据此判断工作量和风险。

需求分解要解决的问题很具体：把模糊愿望变成可设计、可实现、可验证、可追溯的工程条目。比如 ACC 要能跟车只能表达方向。进入项目后，还要回答更多问题：什么时候允许开启，前车怎样定义，跟车距离按什么规则算，谁负责发制动请求，ESP 执行失败怎么办，驾驶员踩刹车以后系统怎样退出，测试时怎样判定通过。

# 系统工程师的工作位置
## 协作角色
系统工程师处在功能、架构、软件、硬件、测试和项目之间。这个岗位不一定亲自写算法代码，也不一定亲自调底盘执行器，但要把整条功能链路想清楚，并把各方责任写成可以执行的需求。

在一个智驾功能项目里，系统工程师经常要和这些角色协同：

| 角色 | 关注点 | 系统工程师需要给出的内容 |
| --- | --- | --- |
| 产品 / 功能定义 | 用户想要什么体验，功能覆盖哪些场景 | 功能目标、使用场景、功能边界 |
| 架构工程师 | 功能由哪些控制器和模块承担 | 系统边界、子系统责任、接口关系 |
| 感知工程师 | 目标、车道、障碍物能否被识别 | 感知输入需求、有效性条件、置信度要求 |
| 规划控制工程师 | 车辆怎样运动，目标速度和目标轨迹怎样生成 | 控制目标、状态机、限制条件 |
| 底盘 / 执行器工程师 | 制动、转向、驱动能否执行请求 | 执行接口、能力边界、反馈语义 |
| HMI 工程师 | 驾驶员看到什么、听到什么、如何接管 | 状态显示、提示语义、交互流程 |
| 测试工程师 | 怎样证明功能满足需求 | 测试场景、通过准则、故障注入项 |
| 功能安全 / SOTIF | 风险、危害、性能不足和误用 | 安全目标、降级策略、ODD / ODC 边界 |

系统工程师要把这些角色的语言翻译到同一套需求体系里。产品说功能要自然，测试需要知道自然对应什么指标；软件说需要输入信号，架构需要知道信号来自哪个控制器；底盘说能力有限，功能需要知道哪些场景必须退出或降级。

## 交付物
需求分解最终会形成一组文档和表格。不同公司命名会有差异，但内容大体相近。

| 交付物 | 作用 |
| --- | --- |
| 功能规范 | 描述功能目标、场景、状态、交互和边界 |
| 系统需求 | 定义整车功能链路需要满足的条件和行为 |
| 子系统需求 | 把系统需求分配给感知、融合、控制、HMI、底盘等对象 |
| 软件需求 | 把子系统行为继续落到软件模块、逻辑条件和参数 |
| 接口需求 / ICD | 定义控制器之间的信号、服务、周期、取值和超时策略 |
| 状态机 | 定义功能状态、进入条件、保持条件、退出条件 |
| 追溯矩阵 | 记录需求来源、下游设计项和测试用例 |
| 测试需求 | 定义验证场景、输入条件、预期输出和通过准则 |

这些交付物之间要能对上。功能规范里写 ACC 支持跟停，系统需求里就要有跟停条件和退出条件；接口需求里要有驻停、起步或制动相关信号；测试需求里要有前车减速至停车、短暂停车再起步、长时间驻停退出等场景。

# 需求分解的基本概念
## 需求是什么
需求是对系统必须满足的条件、行为、性能或约束的描述。它需要比想法清单和设计方案清单更严格。合格需求要能回答几个问题：
1. 谁承担这个需求。
2. 在什么条件下生效。
3. 系统要做什么。
4. 做到什么程度。
5. 输出给谁。
6. 怎样验证。

例如 <font color="#00b050">系统应平顺减速</font> 这个表达太宽。它没有说明系统是谁、什么场景、平顺怎样判断。可改写为：<font color="#00b050">当 ACC 处于激活态且前车减速导致实际时距低于设定时距时，ADAS 控制器应生成纵向减速度请求，并使目标加速度变化率不超过标定阈值</font>；该需求通过 HIL 和实车跟车减速场景验证。

## 需求分解是什么
**需求分解就是从上层目标逐层下钻，直到每条需求都能被某个对象承担，并能被某种方法验证**。它的方向是从抽象到具体，从用户语言到工程语言，从功能目标到模块行为。

需求层级如下：

| 层级 | 关注对象 | 典型问题 | 交付物 |
| --- | --- | --- | --- |
| 用户 / 干系人需求 | 驾驶员、主机厂、法规、安全目标 | 用户想要什么，法规和项目要求什么 | 用户需求、功能目标、项目约束 |
| 系统需求 | 整车功能链路 | 功能在什么场景下做什么，边界是什么 | 系统需求规格、功能规范 |
| 子系统需求 | 感知、规划、控制、HMI、底盘执行 | 每个子系统分别承担什么责任 | 子系统需求、接口需求 |
| 软件需求 | 软件组件和算法模块 | 软件逻辑、参数、诊断、容错怎样实现 | 软件需求规格、模块设计输入 |
| 接口需求 | 信号、服务、总线、诊断 | 控制器之间怎样交换信息 | ICD、信号矩阵、服务接口定义 |
| 验证需求 | 测试和确认对象 | 怎样证明需求已经满足 | 测试用例、仿真用例、HIL / 实车验证项 |

上层需求定义方向，下层需求定义责任、条件、接口和验证。需求分解不追求把文字写得越来越多，而是让<font color="#00b050">每一层都比上一层更可执行</font>。

## 需求、设计和测试的关系
三者有关联，但关注点不同。

| 类型 | 关注问题 | 示例 |
| --- | --- | --- |
| 需求 | 系统必须满足什么 | ACC 激活态下，驾驶员踩下制动踏板时，系统应退出激活态 |
| 设计 | 系统怎样做到 | ACC 状态机监听制动踏板信号，触发 Exit 状态，并清除目标加速度请求 |
| 测试 | 怎样证明做到 | 在 HIL 中使 ACC 进入激活态，注入制动踏板有效信号，检查状态和输出信号 |

需求负责规定目标和约束，设计负责选择实现路径，测试负责验证结果。需求里可以提出接口和状态要求，但**不应过早绑定某个内部实现细节，除非这个实现细节本身就是项目约束**。

# 分解前的输入材料
## 功能输入
开始分解前，要收集能定义功能方向的材料。材料不完整时也可以开工，但需要明确哪些内容已冻结，哪些内容还需要确认。

==功能输入包括：==
1. ==功能目标：例如 ACC、LKA、APA、APO 要解决什么驾驶任务。==
2. ==使用场景：高速巡航、拥堵跟车、自动泊入、自动驶出、驾驶员接管。==
3. ==目标用户：驾驶员、标定工程师、售后诊断人员、测试人员。==
4. ==市场和法规：法规限制、区域差异、NCAP 或主机厂评分要求。==
5. ==竞品或平台约束：已有平台能力、复用控制器、复用传感器、量产成本。==

这些材料决定功能目标。比如 ACC 如果只面向高速巡航，需求重点会放在中高速跟车和定速巡航；如果支持 Stop&Go，还要补零速跟停、驻停保持、起步提醒、自动恢复和长时间停车退出。

## 系统输入
系统输入决定功能能落到哪些工程对象上。

系统输入包括：
1. 控制器架构：ADAS、ESP、VCU、EPS、BCM、IPC、HU 分别有哪些能力。
2. 传感器配置：摄像头、毫米波雷达、超声波雷达、环视、定位模块。
3. 执行器能力：制动能力、转向能力、驱动扭矩响应、挡位控制权限。
4. 通信架构：CAN、CAN FD、以太网、DDS、服务接口、诊断链路。
5. 诊断机制：DTC、DID、故障等级、故障恢复策略。
6. 安全机制：功能安全、安全目标、ASIL 分解、SOTIF 场景约束。

如果不知道系统架构，需求很容易写成空泛愿望。比如 系统应自动制动 这句话必须落到 ADAS 输出请求、ESP 执行制动、VCU 协调扭矩、HMI 提示驾驶员 这条链上，才能进入设计。

## 验证输入
验证输入会反过来影响需求写法。测试条件无法构造、结果无法观测、通过标准无法判断的需求，后期一定会返工。

验证输入包括：
1. 仿真能力：场景库、车辆模型、传感器模型、交通参与者模型。
2. SIL / HIL 能力：控制器闭环、接口注入、故障注入、信号记录。
3. 台架能力：底盘执行器、传感器台架、通信一致性测试。
4. 实车资源：道路、场地、目标车、软目标、天气条件、测试里程。
5. 数据分析能力：日志字段、采样周期、评价脚本、指标计算方法。

需求写作时要考虑验证路径。例如 前车切入后系统应及时减速 缺少可验证性；改成 前车在本车道前方 `30 m` 以内切入，目标置信度满足阈值且 ACC 处于激活态时，系统应在标定时间内生成减速度请求，测试就可以通过仿真、HIL 或场地试验构造。

# 分解流程
## 第一步：定功能边界
功能边界回答功能管什么、不管什么、依赖什么。边界不清时，后续所有需求都会互相挤压。

以 ACC 为例，边界可以这样写：
1. ACC 负责纵向辅助控制，包括定速巡航、跟车巡航和部分 Stop&Go 场景。
2. ACC 不负责横向车道保持。
3. ACC 不把行人作为启动时的跟车目标。
4. ACC 依赖前向感知、ESP、VCU、HMI 和驾驶员监督。
5. ACC 在传感器不可用、ESP 纵向控制不可用或驾驶员接管时退出。

**边界要尽量早确认**。边界一旦变化，下游状态机、接口、HMI、测试用例都会跟着变化。

## 第二步：拆场景
场景是需求分解的入口。**智驾功能很少能用一条规则覆盖所有情况，必须按场景拆开**。

ACC 可拆成以下场景：
1. 无前车定速巡航。
2. 稳定跟车巡航。
3. 前车减速。
4. 前车停车。
5. 前车再起步。
6. 前车切入。
7. 前车切出。
8. 驾驶员踩制动接管。
9. 驾驶员踩油门临时接管。
10. 传感器或执行器故障。

每个场景都要继续写清条件、系统行为、输出接口和验证方法。例如 前车停车 这个场景至少要回答：停车前系统怎样减速，停车后是否保持驻停，前车几秒内起步时是否自动跟随，超过时间后是否提示驾驶员确认，驾驶员开门或解安全带时怎样退出。

## 第三步：建状态机
状态机描述功能处于什么状态，以及状态之间怎样跳转。没有状态机，需求会散成一堆条件句，后面很难处理冲突。

ACC 的状态可简化成几类：

| 状态 | 含义 | 示例条件 |
| --- | --- | --- |
| `Off` | 功能关闭 | 驾驶员未开启，或系统关闭 |
| `Unavailable` | 功能不可用 | 传感器故障、ESP 不可用、车辆条件不满足 |
| `Standby` | 功能待命 | 条件满足，但未进入控制 |
| `Active` | 功能激活 | 系统正在输出纵向控制请求 |
| `Override` | 驾驶员临时接管 | 驾驶员踩油门，系统暂时让出纵向控制 |
| `Exit` | 退出 | 驾驶员取消、踩制动、故障或边界失效 |
| `Failure` | 故障 | 关键故障导致功能不可恢复 |

状态机需求要写清四类条件：
1. 进入条件：怎样从一个状态进入另一个状态。
2. 保持条件：停留在当前状态需要满足什么。
3. 退出条件：哪些事件会触发退出。
4. **优先级：多个事件同时发生时先处理谁。**

例如 ACC 激活态下，驾驶员踩制动和前车减速同时发生时，应优先处理驾驶员接管，退出 ACC 激活态，并停止向 ESP 输出目标加速度请求。这里的优先级比单纯写系统应退出更有工程价值。

## 第四步：拆子能力
功能边界和状态机确定以后，就可以把功能拆成子能力。这里的子能力指系统行为里的责任块，和组织架构里的部门划分没有必然对应关系。

ACC 子能力如下：

| 子能力 | 负责内容 | 典型输出 |
| --- | --- | --- |
| 可用性判断 | 判断车辆、传感器、执行器和驾驶员状态是否允许使用 ACC | ACC 可用 / 不可用 |
| 设定值管理 | 管理设定速度、时距档位、恢复速度 | 目标巡航速度、目标时距 |
| 主目标管理 | 选择前方有效目标并维护目标 ID | 主目标 ID、目标有效标志 |
| 纵向决策 | 根据前车和设定值生成控制目标 | 目标加速度、目标速度 |
| 执行协同 | 与 ESP / VCU 交互，实现制动和驱动 | 制动请求、扭矩协同请求 |
| HMI 管理 | 显示状态、设定值、告警和退出原因 | 图标、文字、声音提示 |
| 故障处理 | 处理信号超时、无效值、执行异常和内部故障 | 降级、退出、DTC |

拆子能力的目的，是把系统需求分配给明确对象。比如跟车距离保持需要多个对象协同完成，会拆到主目标管理、纵向决策、执行协同和 HMI 状态显示。

## 第五步：分配责任
责任分配解决 谁做什么。智驾系统经常跨多个控制器，一个需求如果没有责任分配，很容易出现各方默认别人负责的情况。

以 ACC 减速为例：

| 责任对象 | 应承担的内容 |
| --- | --- |
| ADAS 控制器 | 计算目标加速度，输出纵向控制请求 |
| ESP | 接收目标加速度，执行制动控制，反馈执行状态 |
| VCU | 根据需要协调驱动扭矩，处理动力系统可用性 |
| IPC / HU | 显示 ACC 状态、设定值和退出原因 |
| 驾驶员 | 在系统提示或异常情况下接管车辆 |

责任分配还要处理主从关系。例如 ADAS 请求减速，ESP 执行制动；ESP 反馈不可用时，ADAS 应退出 ACC 或进入降级。需求里要写清主控制方、执行方、反馈方和异常处理方。

## 第六步：写接口
接口需求把控制器之间的协作落成信号或服务。没有接口需求，系统需求无法进入软件集成。

==接口需求要写这些信息：==
1. ==信号名称或服务名称。==
2. ==发送方和接收方。==
3. ==周期或触发方式。==
4. ==数据类型和取值范围。==
5. ==物理含义和单位。==
6. ==初始值、无效值、默认值。==
7. ==超时处理。==
8. ==故障处理。==
9. ==与状态机的关系。==

例如 ACC 向 ESP 输出目标加速度，接口需求需要写到信号语义和异常处理，不能停留在 ADAS 发送目标加速度 这一层。更完整的接口需求应包括：ADAS 控制器在 ACC 激活态下周期发送目标加速度请求；目标加速度单位为 `m/s^2`；超出范围时按饱和值处理；信号超时后 ESP 应退出纵向控制；ADAS 收到 ESP 不可用反馈后应退出 ACC 激活态并通知 HMI。

## 第七步：补非功能需求
**非功能需求描述功能做得怎么样**。智驾功能里，非功能需求经常比功能开关更影响体验和安全。

非功能需求包括：
1. 时延：从输入变化到输出响应的最大时间。
2. 精度：速度、距离、角度、轨迹误差。
3. 平顺性：加速度、加加速度、转角变化率。
4. 可用性：功能在指定 ODD 内的可用条件。
5. 鲁棒性：信号抖动、短时丢帧、目标短时丢失时的处理。
6. 诊断性：故障记录、DTC、冻结帧、故障恢复。
7. 安全性：故障后的退出、降级和驾驶员接管提示。

例如 ACC 跟车需求里，功能行为是保持设定时距；非功能需求会继续约束目标加速度变化率、制动响应时间、跟车距离误差、目标短时丢失后的保持策略。

## 第八步：写验证准则
每条需求都应能被验证。验证准则要写清测试条件、输入、预期输出和通过标准。

示例：

| 需求 | 验证准则 |
| --- | --- |
| ACC 激活态下驾驶员踩制动应退出 | 在 HIL 中使 ACC 进入激活态，注入制动踏板有效信号，检查 ACC 状态在标定时间内变为退出态 |
| ACC 应支持稳定跟车 | 实车或仿真中设置前车以 `80 km/h` 匀速行驶，本车 ACC 激活后应稳定保持设定时距 |
| APO 检测到路径内障碍物应制动 | 在泊车场景中向规划路径注入障碍物，检查系统请求制动并进入暂停或退出状态 |
| HMI 应显示不可用原因 | 注入 ESP 不可用信号，检查 IPC / HU 显示 ACC 不可用并给出一致原因 |

验证方法按需求类型选择。逻辑状态跳转适合 HIL，舒适性和平顺性需要实车或高保真仿真，通信超时适合台架，需求一致性可以通过评审和静态检查完成。

# 需求条目的写法
## 基本结构
一条需求可按 <font color="#00b050">条件 + 对象 + 行为 + 约束 + 输出 + 验证</font> 的结构写。句式如下：

<font color="#00b050">在某条件下，某系统 / 模块应执行某行为，并满足某约束。</font>

示例：
1. 当 ACC 处于激活态且驾驶员踩下制动踏板时，ADAS 控制器应在标定时间内退出 ACC 激活态，并停止向 ESP 输出目标加速度请求。
2. 当 ACC 处于激活态且前方存在有效主目标时，ADAS 控制器应根据主目标距离、相对速度、本车速度和时距档位生成目标加速度。
3. 当 APO 处于运行态且路径安全包络内出现障碍物时，ADAS 控制器应请求 ESP 制动，并进入暂停或退出状态。

这样写能保留责任、条件和行为，后续设计和测试也能继续接下去。

## 弱需求和可用需求
需求评审里经常会看到一些<font color="#00b050">弱需求</font>。弱需求的主要问题是<font color="#00b050">缺少工程信息</font>，和文字长短没有直接关系。

| 弱需求 | 问题 | 可用写法 |
| --- | --- | --- |
| ACC 应及时退出 | 缺少触发条件、退出时间、输出动作 | 当驾驶员踩下制动踏板且 ACC 处于激活态时，ACC 应在标定时间内退出激活态，并向 HMI 输出驾驶员接管提示 |
| 系统应平顺减速 | 缺少平顺性指标 | ACC 输出目标加速度时，应限制目标加速度变化率不超过标定阈值 |
| APO 应避免碰撞 | 缺少障碍条件和系统行为 | APO 运行过程中，当障碍物进入规划路径安全包络内时，系统应请求制动并进入暂停或退出状态 |
| HMI 应提示清楚 | 缺少状态和提示语义 | 当 ACC 因 ESP 故障不可用时，IPC 应显示 ACC 不可用状态，并给出与故障原因一致的提示语义 |

弱需求在项目早期可以作为讨论入口，但不能直接作为开发和测试基线。进入基线前，需要补齐条件、对象、行为、约束和验证方式。

## 需求编号和追溯
需求进入项目后要编号。编号服务于评审、变更、测试和问题定位。

编号方式示例：

| 编号 | 含义 |
| --- | --- |
| `ACC-SYS-001` | ACC 系统需求第 1 条 |
| `ACC-SW-012` | ACC 软件需求第 12 条 |
| `ACC-ICD-005` | ACC 接口需求第 5 条 |
| `ACC-VER-020` | ACC 验证需求第 20 条 |

编号之后，每条需求都应记录来源和下游关系。例如 `ACC-SYS-010` 来源于功能目标 跟车巡航，下游分解到主目标管理、纵向控制、ESP 接口和 HMI 显示，再对应若干 HIL 和实车测试用例。需求变更时，影响范围就能顺着追溯关系找到。

# ACC 分解示例
## 从功能目标开始
功能目标可以写成工程上能接受的系统描述：<font color="#00b050">ACC 在驾驶员监督下，根据驾驶员设定速度、设定时距、本车状态和前向目标状态，实现定速巡航和跟车巡航，并在驾驶员接管、系统故障或运行条件不满足时退出</font>。

这句话已经比 ACC 要能跟车 更具体，因为它包含了四类信息：
1. 责任对象：ACC。
2. 输入来源：设定速度、设定时距、本车状态、前向目标状态。
3. 主要行为：定速巡航、跟车巡航。
4. 退出触发：驾驶员接管、系统故障、运行条件不满足。

## 系统需求
从功能目标可以拆出系统需求：

| 编号 | 系统需求 |
| --- | --- |
| `ACC-SYS-001` | ACC 仅在 D 档、驾驶员安全带已系、车门关闭、ESP 可用、前向感知无相关故障时允许进入待命态 |
| `ACC-SYS-002` | 当驾驶员发出激活指令且车辆条件满足时，ACC 应从待命态进入激活态 |
| `ACC-SYS-003` | ACC 激活态下，若前方无有效主目标，系统应按照设定巡航速度进行纵向控制 |
| `ACC-SYS-004` | ACC 激活态下，若前方存在有效主目标，系统应按照设定时距进行跟车控制 |
| `ACC-SYS-005` | ACC 激活态下，驾驶员踩下制动踏板时，系统应退出激活态 |
| `ACC-SYS-006` | ACC 退出时，应停止发送有效纵向控制请求，并向 HMI 输出退出原因 |

系统需求描述整条功能链路，不急着写具体代码。比如 `ACC-SYS-004` 还没有说明主目标怎么选、目标加速度怎么算、ESP 信号叫什么；这些内容会继续分解到子系统和软件需求。

## 子系统需求
系统需求继续下钻后，会分到不同子系统。

| 系统需求 | 子系统 | 子系统需求 |
| --- | --- | --- |
| `ACC-SYS-004` | 感知融合 | 感知融合应输出主目标距离、相对速度、目标 ID 和置信度 |
| `ACC-SYS-004` | 主目标管理 | 主目标管理模块应根据车道关系、距离、速度和置信度维护 ACC 主目标 |
| `ACC-SYS-004` | 纵向决策 | 纵向决策模块应根据本车速度、主目标距离、相对速度和时距档位生成目标加速度 |
| `ACC-SYS-004` | ESP 接口 | ADAS 控制器应向 ESP 周期发送目标加速度请求 |
| `ACC-SYS-004` | HMI | HMI 应显示 ACC 跟车状态、设定速度和时距档位 |

这一步的关键是<font color="#00b050">责任分配</font>。跟车巡航需要感知提供目标，决策生成控制目标，ESP 执行制动，HMI 告诉驾驶员当前状态。

## 软件需求
软件需求会继续细化到模块内部逻辑。

| 子系统 | 软件需求示例 |
| --- | --- |
| 主目标管理 | 当融合目标位于本车道前方且置信度高于标定阈值时，主目标管理模块应将该目标纳入 ACC 候选目标集合 |
| 主目标管理 | 当当前主目标连续丢失超过标定时间时，主目标管理模块应取消该主目标有效标志 |
| 纵向决策 | 当 ACC 处于跟车控制时，纵向决策模块应根据设定时距计算目标跟车距离 |
| 纵向决策 | 当实际距离小于目标跟车距离时，纵向决策模块应生成减速度请求 |
| 状态管理 | 当驾驶员踩下制动踏板时，状态管理模块应触发 ACC 退出 |

软件需求应避免写成伪代码。它要规定模块行为和约束，但算法实现细节可以放到软件设计文档里。例如具体控制器是 PID、MPC 还是查表加限幅，通常属于设计层面的内容，**除非项目明确要求在需求层锁定算法形式**。

## 接口需求
接口需求把控制器之间的协作写清楚。

| 接口 | 需求示例 |
| --- | --- |
| ADAS -> ESP | ADAS 控制器在 ACC 激活态下应周期发送目标加速度请求 |
| ESP -> ADAS | ESP 应周期反馈纵向控制可用状态和执行状态 |
| ADAS -> HMI | ADAS 控制器应输出 ACC 状态、设定速度、时距档位和退出原因 |
| BCM / ACU -> ADAS | BCM / ACU 应提供车门、安全带和驾驶员相关状态 |
| 感知 -> ADAS | 前向感知应输出目标距离、相对速度、目标 ID 和目标有效标志 |

接口需求还必须定义异常情况。比如目标加速度信号超时以后，ESP 应退出纵向控制；ESP 不可用反馈有效以后，ADAS 应退出 ACC 激活态；HMI 收到退出原因以后，应显示和状态一致的提示。

## 验证需求
验证需求把前面的系统需求变成测试用例入口。

| 需求 | 验证场景 | 通过准则 |
| --- | --- | --- |
| `ACC-SYS-001` | D 档、安全带、车门、ESP、传感器条件组合测试 | 仅当全部条件满足时 ACC 进入待命态 |
| `ACC-SYS-003` | 无前车定速巡航 | 本车速度收敛到设定速度附近，状态显示为巡航 |
| `ACC-SYS-004` | 稳定前车跟车 | 本车维持设定时距，目标加速度变化满足平顺性要求 |
| `ACC-SYS-005` | 激活态踩制动 | ACC 在标定时间内退出，纵向控制请求失效，HMI 显示接管或退出 |
| `ACC-SYS-006` | ESP 故障注入 | ACC 退出激活态，HMI 显示不可用或故障相关提示 |

到这里，一条 ACC 功能目标已经从用户能感知的行为，拆到了状态、模块、接口和测试。

# APO 分解示例
## 功能差异
APO 自动驶出和 ACC 的分解重点不同。ACC 主要是行车纵向控制，APO 是低速泊车场景下的横纵向协同控制。它更依赖车位来源、历史泊入轨迹、近距离障碍物、驾驶员确认和底盘低速执行。

APO 的功能目标可以写成：系统在车辆由 APA 自动泊入平行车位后，根据历史泊入轨迹、周边障碍物和驾驶员确认，规划并执行驶出路径，使车辆低速驶出车位，并在驾驶员接管、空间不足、障碍物风险或系统故障时暂停或退出。

## 分解链路
APO 可按下面的链路拆分：

| 层级 | 示例内容 |
| --- | --- |
| 功能目标 | 系统辅助车辆从已自动泊入的平行车位驶出 |
| 场景前提 | 当前车位由 APA 泊入产生，车辆静止，周边空间满足驶出条件 |
| 状态需求 | 系统应包含初始化、待命、预运行、运行、暂停、完成、退出和故障状态 |
| 感知需求 | 超声波雷达应提供车身周边障碍物距离和方位信息 |
| 规划需求 | 路径规划模块应基于泊入轨迹和障碍物位置生成驶出路径 |
| 横向执行需求 | ADAS 控制器应向 EPS 输出目标转角或转向控制请求 |
| 纵向执行需求 | ADAS 控制器应向 ESP 输出速度限制、停车点和制动模式 |
| HMI 需求 | HU / IPC 应提示开始、松开制动、运行中、暂停、完成、取消和故障 |
| 验证需求 | 测试应覆盖空间充足、空间不足、障碍物突入、驾驶员踩刹暂停、故障退出 |

这个例子说明需求分解要贴合功能类型。ACC 的关键在纵向跟驰、目标选择和执行平顺性；APO 的关键在泊入历史、低速空间几何、障碍约束和 HMI 交互。直接套用同一张需求表会漏掉功能特有的边界。

# 质量检查
## 可验证性
需求写完以后，第一项检查是**可验证性**。测试团队拿到需求后，应能判断用什么环境测、输入怎么构造、输出看什么、通过标准是什么。

可验证需求具备以下特征：
1. 条件清楚：需求在哪些状态、场景和输入下生效。
2. 行为明确：系统需要输出、改变、保持或禁止什么。
3. 指标可观察：车速、距离、时间、状态、信号值、告警内容可以被记录。
4. 结果可判定：测试通过和失败之间有明确边界。
5. 验证方法匹配：仿真、SIL、HIL、台架、实车、审查各自适用于不同需求。

示例：

| 弱写法 | 可用写法 |
| --- | --- |
| ACC 应及时退出 | 当驾驶员踩下制动踏板且 ACC 处于激活态时，ACC 应在标定时间内退出激活态，并向 HMI 输出驾驶员接管提示 |
| APO 应避免碰撞 | APO 运行过程中，当障碍物进入规划路径安全包络内时，系统应请求制动并进入暂停或退出状态 |
| HMI 应提示清楚 | 当 ACC 因 ESP 故障不可用时，IPC 应显示 ACC 不可用状态，并给出与故障原因一致的提示语义 |

## 一致性
一致性检查主要看需求之间有没有互相冲突。智驾系统里的冲突包括：
1. 状态机冲突：同一个触发条件在不同章节对应不同跳转。
2. 阈值冲突：速度范围、距离阈值、时间阈值在不同文档里数值不一致。
3. 责任冲突：同一个行为同时写给 ADAS、ESP 和 VCU，但没有主从关系。
4. 接口冲突：信号矩阵里的取值范围和功能规范里的语义不一致。
5. HMI 冲突：功能状态、提示音、文字提示和图标状态不一致。

处理一致性问题时，应优先确定主数据源。例如速度边界以参数表为准，信号定义以冻结后的 ICD 为准，状态语义以系统状态机为准。没有主数据源时，需求评审会变成逐条争论。

## 完整性
完整性强调关键链路没有断点，细节填充应围绕这些链路展开。智驾功能至少需要覆盖以下方面：
1. 正常进入条件。
2. 正常运行行为。
3. 驾驶员接管。
4. 暂停、恢复和退出。
5. 故障和降级。
6. 外部系统不可用。
7. 通信超时和信号无效。
8. HMI 提示。
9. 诊断记录。
10. 测试验证。

如果某条功能链路没有对应的退出、故障或验证需求，后续设计会自然补空白，但补出来的逻辑未必和系统意图一致。

## 可追溯性
可追溯性解决的是需求来源和影响范围问题。每条系统需求最好能向上追溯到功能目标、法规、安全目标或项目约束，向下追溯到子系统需求、软件需求、接口需求和测试用例。

典型追溯关系如下：

| 追溯方向 | 示例 |
| --- | --- |
| 用户需求 -> 系统需求 | 驾驶员希望自动跟车 -> ACC 应支持设定时距下的跟车巡航 |
| 系统需求 -> 子系统需求 | ACC 跟车巡航 -> 感知融合输出主目标距离和相对速度 |
| 子系统需求 -> 软件需求 | 主目标管理 -> 目标选择模块维护主目标 ID 和置信度 |
| 软件需求 -> 接口需求 | 目标加速度计算 -> ADAS 向 ESP 输出目标加速度信号 |
| 需求 -> 测试用例 | 跟车减速需求 -> 前车从 80 km/h 减速至 30 km/h 的 HIL / 实车场景 |

追溯关系在需求变更时尤其关键。比如目标车速上限从 `130 km/h` 改到 `135 km/h`，受影响的内容包括功能描述、HMI 显示、参数配置、信号范围、控制限幅、测试用例和用户手册。
