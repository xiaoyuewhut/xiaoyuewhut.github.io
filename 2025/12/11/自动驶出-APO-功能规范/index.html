自动驶出 APO 功能规范

发表于2025-12-11|更新于2025-12-11|自动驾驶
|浏览量:

引言

Autonomous Pull Out，自动从车位驶出功能，属于智能泊车（APA/AVP）体系中的一个子功能。

若车辆是使用 APA（自动泊车） 功能泊入平行车位的，且泊入后车辆与周边车辆、围栏的距离满足一定条件，则可以使用自动驶出功能。

APO 根据上次自动泊车时的行驶轨迹，结合传感器实时探测的周边环境信息规划路径，控制车辆的横向和纵向运动，直至驶出车位。

  • 前提条件：车辆必须是使用 APA 功能自动泊入当前车位的。系统会记录泊入时的轨迹状态。
  • 路径规划：根据上次自动泊车的路径以及当前车身周边的环境信息（超声波+毫米波雷达），计算出驶出路径。驶出方向与上次泊入的方向一致。
  • 控制范围：系统接管车辆的横向（转向）和纵向（速度、换挡）控制，直到车辆驶出车位 。

系统架构

超声波雷达为主，毫米波雷达为辅，APO 复用 ADAS 系统的硬件架构，采用 ADAS控制器 → 执行器（EPS/ESP/VCU） 的控制链路。

感知层配置

对 ADAS 传感器布置不清楚的可以先看 https://xiaoyuewhut.github.io/2025/12/05/ADAS-传感器布置/

  • 超声波雷达 (Ultrasonic Sensor)
    • 数量：12个
    • 通信：通过私有 CAN 将距离数据传输给 ADAS 控制器
    • 参数： 频率 48±2 kHz，水平开角 120°，垂直开角 60°
    • 探测范围：短距模式 2.5m，长距模式 4.5m，分辨率 10cm
  • 前向毫米波雷达 (Millimeter Wave Radar)
    • 数量：1 个
    • 功能：探测前方 150m 范围内的障碍信息
    • 参数：频率 77GHz，探测距离 160m，水平开角 ±45°
  • 前视摄像头 Camera
    • 参数：分辨率 1280*960，综合视距 150m 以上
    • 通信：通过 LVDS 接口连接

执行路径

APO采用 ADAS控制器→EPS/ESP→VCU 的分层控制架构：

  • 横向控制 EPS
    接收 ADAS 发送的转角目标值 ADAS_APTarSteeringAngle

  • 纵向控制 ESP
    接收 ADAS 发送的距离停车点距离 ADAS_RemainDistToStopPoint 及最大速度限制 ADAS_MaxVelocityLimit

  • 档位控制 VCU
    接收换挡请求，执行 R/D/P 档切换

  • 人机交互 HMI
    人机交互 (HMI)：通过 HU（主机）与驾驶员交互，显示状态并接收控制指令 。

注意：HMI 显示的车速与实际车速存在换算关系： * 实际车速 < 25km/h 时：仪表车速 = 实际车速 * 1.15

状态机设计

APO状态机的核心挑战在于如何安全地复用APA的泊入轨迹。定义了7大状态，覆盖从初始化到故障的全生命周期。

状态定义

  1. Initialization (初始化)：功能默认关闭，车辆启动后进入此状态。判断是否满足由 APA 泊入及周边距离条件 。
  2. Standby (待命)：满足所有准入条件，等待驾驶员点击“开始” 。
  3. Prerunning (预运行)：（关键中间态） 驾驶员点击了开始，但尚未松开制动踏板。系统等待踏板松开 。
  4. Running (运行)：系统接管控制，按规划路径自动行驶 。
  5. Pause (暂停)：驾驶员踩下制动踏板（短时），系统暂停，等待“继续”或“取消” 。
  6. Finish (完成)：车辆已驶出车位 。
  7. Quit (退出) / Failure (故障)：功能取消或因系统故障退出 。

什么是驾驶员松开制动踏板与APO开始控制之间的时序竞争问题？

想象一下，你和朋友一起开车，朋友坐在驾驶座，你坐在副驾。你们约定好：朋友松刹车，你立刻帮忙打方向盘把车开出车位。

但这里有个问题：你怎么知道朋友"完全"松开了刹车？

场景 A：理想情况

  • 朋友脚从刹车踏板上移开
  • 你清楚地看到脚移开了，然后你打方向盘
  • 车平稳地开出去

场景 B：竞争情况

  • 朋友脚刚抬起来一半，你以为他已经松开了，立刻开始猛打方向盘
  • 结果：车还刹着呢，方向盘在打，车发出"咯吱咯吱"的声音，轮胎原地摩擦
  • 更危险的是：如果在坡道上，车还可能往后溜，撞到后面的车

这就是APO的时序竞争问题，一边是驾驶员的脚（松刹车这个动作），另一边是APO系统（开始控制转向和油门）。这两边的交接必须严丝合缝，否则就会打架。而且旁边就是别人的车，前后只有20厘米空隙，稍微抖一下就可能刮蹭。

所以才加了一个 Prerunning 状态，就像和朋友约定一个暗号：

  • 朋友脚从刹车移开后，先举起右手示意
  • 你看到示意后，数1、2、3，确认他真的完全松开了
  • 然后你才打方向盘

这个"数1、2、3"的过程就是Prerunning状态，它强制系统等待100毫秒，确认：

  1. 刹车信号确实是"松开"
  2. 刹车液压压力确实降到0
  3. 驱动电机准备好
  4. 周围没有新障碍物

关键信号交互

在 Running 状态下，核心控制信号流如下：

  • ADAS → EPS：发送 ADAS_EPS_APCtrlReq = EPS Control 请求控制，并持续发送 ADAS_APTarSteeringAngle。
  • ADAS → ESP：发送 ADAS_AP_BrakeModeStatus = active，并实时更新 ADAS_RemainDistToStopPoint（距离停车点的曲线距离）
  • ESP 响应：根据剩余距离和转向角计算加速度，保证稳定性

性能指标

APO的性能指标需在厘米级定位精度与乘坐舒适性之间取得平衡，远低于ACC的公里级车速控制精度要求。

指标项 性能要求
支持车位 仅支持平行（行式）车位
车位最小长度 6.25m
换挡次数 ≤ 5次
驶出用时 < 120s (无障碍情况下)
车速控制 ≤ 10km/h，控制精度 1km/h 内
平顺性 最大加速度变化率 ≤ 4m/s³，正常制动减速度 ≤ 2m/s²
场地坡度 < 10°
可跨越障碍 高度 5cm ~ 8cm
完成时距障碍物距离 ≥ 0.5m

人机交互策略

APO的人机交互核心是 “步步确认，防止误操作” ，因为低速场景下驾驶员更容易分心。

功能局限性

基于当前的传感器配置、算法能力以及系统运行设计域（ODD），APO 功能在以下场景中存在固有限制，驾驶员需保持警惕并准备随时接管：

  • 泊车历史记录限制

    • 非 APA 泊入：车辆并非通过 ADAS 的 APA 功能泊入当前车位，APO 功能将不可用。
    • 记忆丢失：如果车辆在泊入后发生断电或系统重启，导致上次 APA 泊入的历史轨迹数据丢失，APO 功能将被禁止使用。
    • 障碍物变动：泊入完成后，若车位周边障碍物（如前后车辆）的位置发生大幅变动，超过系统设定的安全阈值（例如 30cm），APO 将被禁止使用，以防止碰撞风险。

  • 环境与场地限制

    • 车位类型：仅支持平行（行式）车位的驶出。垂直或斜列车位因轨迹规划误差较大，被明确禁用。
    • 场地坡度：当场地坡度大于 10∘10^{\circ}10∘ 时，由于轮胎附着力变化和转向阻力矩增大，轨迹跟踪误差可能超过 15cm，功能将触发退出 (ESP_RoadGradient > 10°)。
    • 障碍物高度：对于高度低于 5cm 或高于 8cm 的障碍物，超声波雷达的探测可靠性下降。低于 5cm 的障碍物可能漏检；高于 8cm 的障碍物可能导致车辆底盘剐蹭。

  • 传感器与环境干扰

    • 恶劣天气：雨、雪、雾等天气导致能见度下降时，前视摄像头和毫米波雷达的性能都会受到影响。
    • 传感器遮挡：超声波探头被泥土、冰块、积雪等遮盖，导致回波信号衰减或错误。
    • 超声波干扰：附近存在其他同频工作的超声波设备（如其他车辆的泊车辅助系统），可能导致 APO 的测距数据不稳定。
    • 动态环境：在行人穿梭频繁或有横穿车辆的复杂动态场景中，为了安全避障，功能可能会频繁触发暂停或被动退出（T8），导致用户体验下降。

  • 系统限制与精度

    • 车速限制：功能仅在低速（≤10km/h\le 10 \text{km/h}≤10km/h）下运行。
    • 定位精度：APO 的纵向控制强调厘米级定位精度，一旦系统误差超过预设的跟踪阈值，将触发退出，要求驾驶员接管。
