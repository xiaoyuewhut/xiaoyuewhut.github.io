CAN 是车载电子系统里最常见的总线协议之一。它把物理层、电气判决、优先级仲裁、错误检测和故障隔离连成了一条完整链路。CAN FD 沿用这条链路，把单帧载荷和有效吞吐继续往上推。

学习 CAN 和 CAN FD 时，经常会遇到两个断点。一个断点在物理层和协议层之间：端接、差分线、位时序看起来像硬件问题，仲裁、ACK、CRC 又像协议问题，但实际通信时它们连在同一条链上。另一个断点在帧结构内部：资料里会列出一串字段名，却没有把字段组和具体位域的关系说清楚。

下面按一帧报文的传输过程整理：报文怎样从节点里发出来，怎样在总线上完成仲裁、传输和确认；这个过程怎样对应到帧结构；CAN FD 改了哪些位置；CAN FD 之后的 CAN XL 又处在什么位置。

# Classical CAN 的通信过程
## 节点组成
一个 CAN 节点通常可以拆成三个层次来看：

| 对象 | 属性 | 主要职责 |
| --- | --- | --- |
| `CAN Controller` | 硬件控制器，常集成在 MCU 或独立通信控制器中 | 位时序、仲裁、帧格式、CRC、ACK、错误计数、邮箱管理 |
| `CAN Transceiver` | 物理层器件 | 逻辑电平和差分总线电平转换、总线驱动、接收整形、电气保护 |
| `CAN Driver / CAN Interface / PduR / COM` | 软件栈 | 报文配置、发送请求、接收分发、与应用层交互 |

这三个对象的边界会影响后面的故障判断。仲裁、CRC、ACK 和 Bus-Off 属于控制器层面的协议行为；波形质量、显隐性判决、总线保护更多取决于收发器、端接和线束；软件决定节点发送哪些报文、发送周期怎样配置、接收内容怎样交给上层模块。

## 物理层前提
CAN 物理层通常使用 `CAN_H` 和 `CAN_L` 两根差分线。控制器最终关心的是总线当前呈现出的逻辑状态，而单根线对地电压只是形成这个状态的电气表现。这个状态可以分成两种：
1. `Dominant`
2. `Recessive`

显性位能够覆盖隐性位。这个电气特性决定了 CAN 可以做位级仲裁。几个节点同时起发时，只要某一位上有节点输出显性，总线最后被所有人读到的就是显性。某个节点如果自己发送的是隐性位，却在总线上读到了显性位，就会立刻知道自己已经失去这次发送权。

这个电气特性直接连到协议层。CAN 的仲裁能够成立，依赖的就是多个节点边发边比，再由总线上的实际位值决定谁继续发送。

## 端接、拓扑和位时序
CAN 总线通常在物理两端各接一个终端电阻，用来做阻抗匹配和抑制反射。主干长度、支路长度、收发器延时和比特率之间也有成组约束。报文字段和控制位属于协议视角，端接、电阻、支路长度属于物理视角；进到实际网络里，这两部分会共同决定一帧报文能不能被稳定采样。

这些约束一旦失配，问题往往会顺着链路层层传导：
1. 波形边沿振铃变大。
2. 采样点附近电平不稳定。
3. 位错误、CRC 错误和 ACK 丢失开始出现。
4. 控制器错误计数持续增加。

CAN 的协议行为需要物理层支撑。物理层余量足够，总线上的每个节点才能在相近的时间点读到相同的位值；只有这个前提成立，仲裁、校验和确认才谈得上可靠。

## 一帧报文的传输链路
把前面的背景放在一起，一帧 Classical CAN 数据报文可以按下面这条链路来理解：

| 阶段 | 总线上发生什么 | 这一阶段解决的问题 |
| --- | --- | --- |
| `Bus Idle` | 总线保持隐性，等待新的发送机会 | 节点判断当前能否起发 |
| `SOF` | 某个节点发出帧起始位 | 所有节点开始同步 |
| `Arbitration` | 多个节点可以同时竞争总线 | 决定哪一帧获得发送权 |
| `Control` | 获胜节点继续给出帧格式和长度信息 | 接收端知道后面要按什么格式解释报文 |
| `Data` | 发送有效载荷 | 传输真正的业务内容 |
| `CRC` | 发送端给出校验序列 | 接收端判断整帧内容是否可靠 |
| `ACK` | 正确接收的节点在 ACK 槽写显性位 | 发送端确认这次通信已被其他节点接收 |
| `EOF` | 发送端结束当前帧 | 一帧逻辑上传输完成 |
| `Intermission` | 总线进入最小帧间隔 | 为下一帧留出边界 |

这组阶段既描述通信过程，也可以作为后面理解帧结构的索引。后面再看到 `SOF`、`Arbitration`、`CRC` 这些字段时，可以把它们放回这条链上看。

## 仲裁、ACK 和位时序
仲裁解决的是 谁先使用总线。CAN 是多主总线，任何节点只要发现总线空闲，都可以尝试起发。真正留下来的那一帧，由仲裁字段逐位竞争决定。发送隐性却读到显性的节点，会立刻停止发送并转为接收状态。因为总线上已经发出的位始终保持有效，这个过程被称为无破坏仲裁。

ACK 检查的是`这帧报文有没有真正到达其他节点`。发送节点不会自己确认自己。只有已经正确收到整帧、并且 CRC 检查通过的节点，才会在 ACK 槽主动把总线拉成显性。发送端如果没有看到 ACK，就会把这次发送记为失败。这类失败有时来自接收端协议校验，有时来自位时序、收发器状态、端接或线束问题。

假设 `ESP ECU` 周期性发送一帧 `0x1A0` 报文，数据里放的是四轮轮速。`VCU`、`ABS` 和仪表都需要这帧报文。发送过程如下：
1. `ESP ECU` 发送 `SOF`，整条总线开始同步。
2. 它继续发 `Identifier = 0x1A0`、控制字段、8 字节轮速数据和 `CRC`。
3. `VCU`、`ABS`、仪表三个节点一边接收，一边各自在本地做位判决、长度检查和 `CRC` 检查。
4. 报文走到 `ACK Slot` 时，`ESP ECU` 自己不会去写显性位，而是释放总线，并在采样点读取总线结果。
5. 只要 `VCU`、`ABS`、仪表里有任意一个节点已经正确收完整帧，它就会在 `ACK Slot` 把总线拉成显性。
6. `ESP ECU` 在采样点读到显性位，就会认为这次发送成功。

多个接收节点同时存在时，所有正确收到该帧的节点都可以 ACK。它们会在同一个 `ACK Slot` 同时把总线拉成显性。这里不会冲突，因为 CAN 总线按显性覆盖隐性的规则工作。对发送端来说，只要这个时刻总线变成显性，ACK 就成立；它并不知道是 `VCU` 在 ACK，还是 `ABS` 和仪表也一起在 ACK。

**ACK 只能证明至少有一个其他节点正确收到了这帧报文，不能证明所有需要它的节点都收到了。**

沿用上面的例子，如果 `VCU` 和仪表正确收到了 `0x1A0`，但 `ABS` 因为采样点配置不合适导致 `CRC` 检查失败，那么 `VCU` 和仪表仍然会在 `ACK Slot` 拉显性，发送端看到 ACK 以后会认为这次发送成功；但对 `ABS` 来说，这帧报文依然是无效的，它后续会按自己的错误处理逻辑响应。

如果总线上只有 `ESP ECU` 自己在线，或者 `VCU`、`ABS`、仪表都因为收发器异常、端接问题、线束故障、位时序不匹配而没有正确收下这帧报文，那么 `ACK Slot` 上就不会有人把总线拉成显性。`ESP ECU` 采样到的仍然是隐性位，于是它会把这次发送判定为 ACK 失败，并进入后续的重发和错误处理路径。

位时序则决定节点在什么时刻采样总线。控制器会按配置好的采样点来判定位值，不会在一个比特周期里任意挑时间读总线。工程上常见的配置参数包括：
1. `Bit Rate`
2. `Sample Point`
3. `Propagation Segment`
4. `Phase Segment`
5. `SJW`

如果节点之间的时钟误差和传播延时没有被这些参数吸收掉，不同节点看到的位边界就会慢慢错开。错开的结果不会停留在抽象层面，最终会体现在仲裁异常、CRC 异常、ACK 异常甚至 Bus-Off 上。

# 再把这个过程对回帧结构
## 整帧骨架和具体位域
帧结构先看字段组，再看字段组内部的位域。Classical CAN 数据帧可以整理成下面这组骨架：

| 位置 | `SOF` | `Arbitration` | `Control` | `Data` | `CRC` | `ACK` | `EOF` | `Intermission` |
| --- | --- | --- | --- | --- | --- | --- | --- | --- |
| 对应阶段 | 起发与同步 | 总线竞争 | 格式说明 | 业务数据传输 | 整帧校验 | 接收确认 | 帧结束 | 帧间隔 |

这个骨架和上一节的通信过程是一一对应的。通信过程描述总线上发生了什么，帧结构描述这些过程在报文格式里落成了哪些字段组。把这两层对上以后，再去区分标准帧、扩展帧和 CAN FD。

## 11-bit 标准数据帧
对于最常见的 11-bit 标准数据帧，可以把字段组继续展开成下面这张表：

| 字段组 | 具体位域 | 说明 |
| --- | --- | --- |
| `SOF` | `SOF` | 宣告一帧开始，并触发同步 |
| `Arbitration` | `Identifier[10:0] + RTR` | 标识报文并完成优先级竞争 |
| `Control` | `IDE + r0 + DLC` | 指定帧格式并给出数据长度码 |
| `Data` | `Data Field` | 最多承载 8 字节应用数据 |
| `CRC` | `CRC Sequence + CRC Delimiter` | 让接收端检查整帧内容是否出错 |
| `ACK` | `ACK Slot + ACK Delimiter` | 由接收端确认报文已被正确接收 |
| `EOF` | `EOF` | 标记当前帧结束 |
| `Intermission` | `Intermission` | 在前后两帧之间留出最小间隔 |

`Identifier` 和 `DLC` 的位置需要单独记一下。`Identifier` 属于 `Arbitration`，因为它直接参与总线竞争；`DLC` 属于 `Control`，因为它描述的是后续数据字段的长度信息。后面看控制器寄存器、DBC 配置或总线抓包时，这层归属关系会反复出现。

## 29-bit 扩展数据帧
扩展帧没有换掉整帧骨架，变化主要集中在仲裁字段内部。它的核心展开形式如下：

| 字段组 | 具体位域 | 说明 |
| --- | --- | --- |
| `SOF` | `SOF` | 帧起始 |
| `Arbitration` | `Base Identifier[10:0] + SRR + IDE + Extended Identifier[17:0] + RTR` | 仲裁字段扩展为 29-bit 标识符路径 |
| `Control` | `r1 + r0 + DLC` | 继续承载控制和长度信息 |
| `Data` | `Data Field` | 最多 8 字节 |
| `CRC` | `CRC Sequence + CRC Delimiter` | 检错 |
| `ACK` | `ACK Slot + ACK Delimiter` | 接收确认 |
| `EOF` | `EOF` | 帧结束 |
| `Intermission` | `Intermission` | 帧间隔 |

扩展帧的重点是理解<u>仲裁字段为什么会变长</u>。

标准帧中，仲裁字段主要由 `11-bit Identifier + RTR` 组成；到了扩展帧，这一组内容被拉长成 `基标识符 + SRR + IDE + 扩展标识符 + RTR`。字段组没有变，变化发生在字段组内部。这个视角能把标准帧和扩展帧自然地连起来。

## 位填充和实际波形
帧结构表给出的是逻辑字段顺序，**总线上真正出现的波形还要再叠加位填充规则**。

<font color="#2DC26B">位填充的作用可以从同步角度理解：CAN 节点需要依靠总线边沿不断校准自己的采样时刻，如果总线上长时间保持同一个电平，接收节点就缺少重新同步的机会。位填充通过人为插入反相位，强制总线上定期出现边沿。</font>

总线边沿有点像跑步时的节拍声。接收节点除了判断高低电平，还要在合适的时刻采样。节拍声隔一段时间出现一次，接收节点就能修正自己的步伐；长时间没有节拍时，接收节点只能靠自己的时钟往前估，时间一长就容易偏。位填充插入的反相位，相当于在连续同电平中补一个节拍，让接收节点有机会重新对齐采样位置。

Classical CAN 的基本规则很直接：从 `SOF` 开始，到 `CRC Sequence` 结束，只要发送端连续发送了 5 个相同位，就会自动插入 1 个相反电平的填充位。这个填充位只是线上传输用的辅助位，不属于 `Identifier`、`DLC`、`Data Field` 或其他任何逻辑字段。接收端按同一条规则识别它，并<font color="#ff0000">在解释报文之前把它删除</font>。

看一个具体比特串。假设某一段原始逻辑内容里连续出现了 6 个 `0`：

| 内容 | 比特串 |
| --- | --- |
| 原始逻辑内容 | `000000` |
| 线上实际发送 | `00000 1(stuff) 0` |
| 接收端还原后 | `000000` |

这里的 `1(stuff)` 是发送端在 5 个 `0` 后面自动插入的填充位，最后那个 `0` 才是原始逻辑内容里的第 6 个 `0`。接收端看到 5 个连续 `0` 以后，会把紧跟着的反相位当作填充位丢掉，再继续解析后面的真实字段。

如果原始逻辑内容里连续出现 6 个 `1`，规则完全一样，只是插入的填充位会变成 `0`：

| 内容 | 比特串 |
| --- | --- |
| 原始逻辑内容 | `111111` |
| 线上实际发送 | `11111 0(stuff) 1` |
| 接收端还原后 | `111111` |

位填充还会参与错误检测。在需要位填充的区域里，如果接收端看到 6 个连续相同位，就会判断为 Stuff Error。比如接收端在 `SOF` 到 `CRC Sequence` 这段范围内看到了 `000000`，说明发送端没有在 5 个 `0` 后插入反相位，或者线上波形已经被干扰破坏。这个节点随后会触发错误处理。

位填充会影响两个现象。<font color="#00b0f0">第一，示波器上看到的连续位数，不会完全等于表格里列出的原始位宽。第二，同样是 8 字节报文，不同位模式下插入的填充位数量可能不同，总线占用时间也会有细微变化。CRC 校验时也要注意这一点：发送端计算 CRC 用的是逻辑内容，接收端会先去掉填充位，再按还原后的逻辑内容做 CRC 检查。</font>

# CAN FD 的扩展位置
## CAN FD 保留的 CAN 思路
CAN FD 保留了 CAN 的基本通信思路，同时对若干关键位置做了增强。它保留的部分主要有三条：
1. 仍然是多主总线。
2. 仍然依赖显性位覆盖隐性位完成仲裁。
3. 仍然按 `SOF -> Arbitration -> Control -> Data -> CRC -> ACK -> EOF -> Intermission` 这条骨架组织整帧。

也正因为这三点没有变，CAN FD 才能被归入 CAN 的扩展体系。

## CAN FD 的来源
Classical CAN 的数据字段最多只有 8 字节。这个长度对很多周期控制报文足够用，但对诊断、刷写、标定和更高密度的数据交换会显得紧张。报文一旦拆得过碎，固定协议开销就会被重复支付，总线效率也会下降。

CAN FD 的扩展目标集中在三点：
1. 单帧载荷扩大到最多 **<font color="#d83931">64 字节</font>**。
2. <font color="#d83931">数据阶段</font>允许使用更高比特率。
3. 在尽量保留原有 CAN 生态的前提下提升有效吞吐。

## CAN FD 的变化集中在 Control、Data 和 CRC
沿着前面的骨架看，CAN FD 的主要改动集中在三组字段：

| 字段组       | 典型变化                      | 变化带来的结果                                                     |
| --------- | ------------------------- | ----------------------------------------------------------- |
| `Control` | 加入 `FDF`、`BRS`、`ESI` 等控制位 | <font color="#d83931">节点能区分 FD 帧、决定是否切换高速数据阶段、反映错误状态</font> |
| `Data`    | 最多从 8 字节扩展到 64 字节         | 单帧承载能力显著提高                                                  |
| `CRC`     | 针对更长载荷和更高相位速率增强           | 在新工作条件下维持足够检错能力                                             |

`Arbitration`、`ACK`、`EOF` 和 `Intermission` 这些部分仍然沿用前面的理解框架。CAN FD 的新增控制位，主要服务于 FD 帧识别、速率切换、错误状态标记和长载荷校验。

## `BRS` 和双速率阶段
CAN FD 最有代表性的变化，是允许一帧内部使用两个速率阶段。仲裁阶段仍然使用较保守的 nominal bit rate，数据阶段在 `BRS = 1` 时可以切换到更高的数据相位速率。

它可以拆成三个阶段：

| 阶段 | 典型内容 | 速率特征 |
| --- | --- | --- |
| `Nominal Phase` | `SOF + Arbitration + Control` 的主要前段 | 所有节点共同参与，速率保持保守 |
| `Data Phase` | `Data Field + CRC` 的主要高速部分 | 由获胜发送者主导，可切高比特率 |
| `Return to Nominal` | `ACK + EOF + Intermission` | 回到多节点共同参与的节奏 |

这里的逻辑和 CAN 原有仲裁机制是连贯的。仲裁阶段需要多个节点一起比较位值，所以速率要稳；数据阶段只剩下一个发送者继续主导，总线具备提速空间；ACK 阶段又需要其他节点参与确认，因此再回到 nominal 域。`BRS` 对应的正是这一段速率切换。

## DLC 映射和物理层敏感度都需要单独记住
CAN FD 还有两点经常在工程里带来误解。

第一，`DLC` 和字节数不再简单线性对应。8 字节以上采用离散映射，常见长度包括 `12`、`16`、`20`、`24`、`32`、`48`、`64`。看到某个 `DLC` 值时，需要按 CAN FD 的映射表确认真实字节数。

第二，CAN FD 对物理层更敏感。高速数据阶段会压缩采样裕量，终端偏差、支路过长、线束反射、收发器延时、节点时钟误差都会更容易暴露成实际故障。一套 Classical CAN 网络能够稳定运行，并不自动意味着切到 CAN FD 高速数据阶段以后仍然稳定。

# CAN FD 之后的 CAN XL
## CAN XL 的位置
CAN FD 之后，CAN 体系里还有 CAN XL。它属于第三代 CAN 数据链路层协议。按照 CAN in Automation 的说法，CAN XL 支持 Classical CAN、CAN FD 和 CAN XL 三类协议形态；Bosch 也把 CAN XL 描述为 CAN 演进里的下一步，用来填补 CAN FD 和 `100BASE-T1` 车载以太网之间的带宽空档。

三代 CAN 的关系如下：

| 代际  | 名称                     | 单帧数据字段                                  | 学习重点                   |
| --- | ---------------------- | --------------------------------------- | ---------------------- |
| 第一代 | Classical CAN / CAN CC | 最多 8 字节                                 | 多主总线、无破坏仲裁、错误处理        |
| 第二代 | CAN FD                 | 最多 64 字节                                | 数据阶段提速、载荷扩大、CRC 增强     |
| 第三代 | CAN XL                 | <font color="#d83931">最多 2048 字节</font> | 更大载荷、更高吞吐、面向骨干网和上层协议承载 |

这组关系体现了 CAN 的演进方向：CAN FD 主要解决 8 字节载荷和有效吞吐不足的问题；CAN XL 继续把载荷扩大到 2048 字节，并把协议设计进一步靠近骨干网、子骨干网和高层协议承载场景。

## CAN XL 主要增强了什么
CAN XL 的变化主要集中在三点。

第一，<font color="#d83931">数据字段大幅增加</font>。ISO 11898-1:2024 的公开摘要中提到，使用 XL 帧格式时，CAN 数据链路层的数据字段可以达到 2048 字节。这个量级已经明显不同于 CAN FD 的 64 字节，适合更大的数据块传输。

第二，<font color="#d83931">数据速率继续提高</font>。Bosch 资料给出的 CAN XL 净数据速率可到 `20 Mbit/s`，并说明 CAN XL 仍然保留 CAN 协议里无破坏仲裁这类优势。它继续沿用 CAN 的基本访问思想，同时把数据阶段能力往上推。

第三，<font color="#d83931">标识和过滤思路更适合复杂网络</font>。CiA 资料提到，CAN XL 把 CAN-ID 相关功能拆成 `11-bit Priority ID` 和 `32-bit Acceptance Field`。前者更偏优先级，后者更偏接收过滤和上层寻址信息。这比单纯扩大原有 ID 更适合复杂网络里的路由、网关和高层协议映射。

## CAN XL 和以太网、CAN SIC 的关系
CAN XL 的定位经常会和车载以太网一起讨论。它主要填补 CAN FD 和以太网之间的空档。对于一些不需要完整以太网交换体系、但又觉得 CAN FD 载荷和吞吐不够的网络，CAN XL 提供了一个延续 CAN 工程生态的选项。（*但是我没看到有什么应用*）

CAN SIC 和 CAN FD-SIC 则更适合归入物理层增强。它们主要改善高速 CAN / CAN FD 网络上的信号完整性和物理层表现。协议代际上，Classical CAN 之后是 CAN FD，CAN FD 之后看 CAN XL；物理层增强上，可以再单独看 CAN SIC、CAN FD-SIC 和 CAN SIC XL。
