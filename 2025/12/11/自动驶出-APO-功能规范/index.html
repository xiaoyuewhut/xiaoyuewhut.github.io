自动驶出 APO 功能规范

发表于2025-12-11|更新于2025-12-11|自动驾驶
|浏览量:

引言

Autonomous Pull Out，自动从车位驶出功能，属于智能泊车（APA/AVP）体系中的一个子功能。

它的目标是：

在低速、驾驶员不接管的前提下，让车辆自动从车位中驶出到可直行的车道上。

在功能分类上，它对应泊车控制中的“出库”功能，但比传统 RPO（Remote Pull Out）/ PPO（Phone Pull Out）更智能。

大部分 OEM 的 APO 流程类似如下：

  1. 车位识别
    系统识别车辆当前处于车位中，确定车位类型（侧方/垂直/斜列）。

  2. 环境建图
    感知模块获取左右车辆边界、墙面、行人等场景要素。

  3. 规划可行驶出路径（Pull-Out Path）
    根据空间情况选择合适的驶出策略，例如：

  • 前进—倒车—前进（多段）
  • 单方向倒车出库
  • 前进“切”出
  1. 安全监测（Safety Monitoring）
  • 摄像头检测障碍物
  • 超声波测距
  • 环视盲区检查
  • 行人/小孩检测

  1. 执行控制（Lateral + Longitudinal）
    自动打方向、控制油门和刹车完成出库。

  2. 达成驶离成功点
    车头进入车道中心 ± 允许偏差，车辆恢复给驾驶员接管。

  3. 功能结束
    APO 状态转入 Finished。

系统架构

超声波雷达为主，毫米波雷达为辅，与 APA 共享传感器套件，但是在算法逻辑层面会专注于狭窄空间的路径逆向复现。

感知层配置矩阵

对 ADAS 传感器布置不清楚的可以先看 https://xiaoyuewhut.github.io/2025/12/05/ADAS-传感器布置/

  • 超声波雷达：12探头方案，48±2kHz工作频率，通过私有CAN接入。前4/后4 短距模式(2.5m)用于车位边界检测，侧4 长距模式(4.5m)用于驶出路径预扫描。水平120°/垂直60°的开角确保对低矮路沿和柱子的探测能力
  • 前向毫米波雷达：77GHz频段，160m探测距离，在APO中主要用于检测垂直于车身方向的横穿车辆/行人，弥补超声波雷达在动态目标识别上的不足
  • 前视摄像头：1280×960分辨率，虽在APO中不直接参与路径规划，但用于记录泊入时的车位标线信息，为驶出方向提供先验约束

执行路径

APO采用 ADAS控制器→EPS/ESP→VCU 的分层控制架构：

  • 横向控制
    ADAS通过ADAS_EPS_APCtrlReq信号请求EPS进入Active状态，发送ADAS_APTarSteeringAngle目标转角，EPS以50ms周期响应。转向速率被限制在450°/s以内，防止轮胎刮蹭路沿

  • 纵向控制
    ADAS向ESP发送ADAS_AP_BrakeModeStatus = active，ESP的CDD(Creep Drive-away Damp)模块接管车速控制。不同于ACC的舒适制动，APO的纵向控制强调厘米级定位精度，ESP根据ADAS_RemainDistToStopPoint实时计算减速度曲线，确保在换挡点前0.15m精确刹停

  • 档位控制
    ADAS通过ADAS_GearsTarget请求VCU执行R→D档切换，VCU反馈VCU_CH_GearControlFeedback确认档位到达。每次换档耗时约0.8s，期间ESP触发保压防止溜车

核心信号带宽与时序

  • 超声波数据刷新率：
    20Hz，12个探头的距离数据通过ADAS_DISTANCE_XXX_Distance16个信号分时发送，每个信号带Rolling Counter和Checksum

  • 控制指令周期：
    ADAS→EPS的ADAS_APTarSteeringAngle每10ms发送一次，EPS反馈EPS_SteeringAngle延迟不超过20ms

  • 状态机同步：
    座舱域控制器与ADAS的ADAS_APOFuncStatus状态信号采用 “发送-确认” 机制，座舱域控制器在收到状态变更后100ms内必须更新HMI显示，避免状态不一致导致用户误操作

状态机设计

APO状态机的核心挑战在于如何安全地复用APA的泊入轨迹。定义了7大状态，覆盖从初始化到故障的全生命周期。

Initialization

KL15上电后，ADAS从EEPROM读取LastAPAStatus标志位，确认车辆是否通过APA泊入。同时检测12个超声波探头的自诊断信号UCM_XXX_SensorFault

此状态下ADAS会持续计算车身与周边障碍物的可驶出裕度

  • 如果是平行泊车场景，车头/车尾距前车/后车的最小距离需≥0.5m（MinClearance_Front ≥ 50cm && MinClearance_Rear ≥ 50cm）

  • 如果检测到泊入后障碍物位置变动超过30cm，置位APO_Available = Unavailable，也就是 APO 会被禁止使用，以防止碰撞风险

若连续3个周期收不到任何超声波数据，直接跳转到Failure状态

Standby

需同时满足9项硬条件：

  • 人机交互：中控屏显示灰色APO图标，并弹出模态对话框 “检测到可驶出车位，是否开始？”，此对话框必须在驾驶员操作后5s内自动消失，防止分心驾驶

  • 动态监控：在Standby状态下，ADAS仍以10Hz频率监控周边障碍物。若检测到侧方车辆以>5km/h速度靠近且TTC<3s，主动取消Standby状态，避免驾驶员启动APO时发生碰撞

Prerunning

这个状态是为了解决驾驶员松开制动踏板与APO开始控制之间的时序竞争问题。

进入Prerunning后，ADAS发送ADAS_AP_BrakeModeStatus = Standby，EPS保持EPS_APStatus = Reply for Control（转角预加载状态）。此时HMI显示 “请松开制动踏板” 浮窗，并倒计时10s。若超时未松开，退回Standby。

  • 坡道处理：当ESP_RoadGradient > 3%时，ESP主动介入CDD保压，防止溜车。此时即使驾驶员松开制动，车辆仍保持静止，直到APO发送ADAS_RemainDistToStopPoint > 0。

什么是驾驶员松开制动踏板与APO开始控制之间的时序竞争问题？

想象一下，你和朋友一起开车，朋友坐在驾驶座，你坐在副驾。你们约定好：朋友松刹车，你立刻帮忙打方向盘把车开出车位。

但这里有个问题：你怎么知道朋友"完全"松开了刹车？

场景 A：理想情况

  • 朋友脚从刹车踏板上移开
  • 你清楚地看到脚移开了，然后你打方向盘
  • 车平稳地开出去

场景 B：竞争情况

  • 朋友脚刚抬起来一半，你以为他已经松开了，立刻开始猛打方向盘
  • 结果：车还刹着呢，方向盘在打，车发出"咯吱咯吱"的声音，轮胎原地摩擦
  • 更危险的是：如果在坡道上，车还可能往后溜，撞到后面的车

这就是APO的时序竞争问题，一边是驾驶员的脚（松刹车这个动作），另一边是APO系统（开始控制转向和油门）。这两边的交接必须严丝合缝，否则就会打架。而且旁边就是别人的车，前后只有20厘米空隙，稍微抖一下就可能刮蹭。

所以才加了一个 Prerunning 状态，就像和朋友约定一个暗号：

  • 朋友脚从刹车移开后，先举起右手示意
  • 你看到示意后，数1、2、3，确认他真的完全松开了
  • 然后你才打方向盘

这个"数1、2、3"的过程就是Prerunning状态，它强制系统等待100毫秒，确认：

  1. 刹车信号确实是"松开"
  2. 刹车液压压力确实降到0
  3. 驱动电机准备好
  4. 周围没有新障碍物

Running

这是APO的核心工作状态，分为轨迹跟踪和动态重规划的双线程：

  • 轨迹复现：ADAS从EEPROM读取APA泊入时记录的Path_X[200]、Path_Y[200]航点序列，以20ms周期进行逆向 spline 插值，生成驶出轨迹。

  • 动态避障：超声波雷达以20Hz扫描车身周边，若检测到新障碍物进入SafetyZone（距车身0.5m），触发ADAS_APWarning = Obstacle Detected，HMI发出高频警示音(800Hz, 200ms间隔)。若障碍物距离<25cm（Range1），立即触发退出条件，ESP执行Emergency制动。

Pause

Running状态下，驾驶员踩下制动踏板且持续时间<6s（IB_BrakePedalStatus = Pressed && PressedDuration < 6000ms），就会触发 Pause。

ADAS发送ADAS_APTarSteeringAngle = EPS_SteeringAngle(保持当前转角)，ESP收到ADAS_RemainDistToStopPoint = 0cm后进入保压。HMI弹出 “继续/取消” 按钮，并显示暂停原因图标。

驾驶员点击继续且松开制动踏板后，APO不立即恢复Running，而是先进入Prerunning确认驾驶员已完全松开制动，防止"制动-油门"竞争，这是一种 Resume 机制。

Quit

退出分两种：

  • 驾驶员主动退出：
    HU发送HU_APOControl = Disable或HU_SuspendFollowupAction = Quit，ADAS发送ADAS_AP_FailureBrakeMode = Comfortable，ESP以-2m/s²减速度平滑停车

  • 安全被动退出：
    检测到转向干预(EPS_EPSTouqbar > 3Nm > 200ms)、油门超限(VCU_CH_AccelPedalPosition > 5%)、车门打开等，ADAS发送ADAS_AP_FailureBrakeMode = Emergency，ESP以-5m/s²紧急制动

车辆静止后，ESP反馈ESP_APLCStatus = standby，ADAS发送ADAS_GearsTarget = P，VCU执行挂P档并拉起EPB。

Failure

采用三级故障管理：

  • Level 1 (Reversible)：单个超声波探头故障，APO功能仍可用，但HMI显示"泊车辅助精度降低"
  • Level 2 (Irreversible)：毫米波雷达故障或EPS_ErrorFlag = LightFailure，APO功能禁能，但APA、BSD等未受影响
  • Level 3 (System Level)：ADAS_ECUFaultStatus = Failure或ESP_ESPFault = Failure，所有ADAS功能熔断，IPC显示红色"高级自动驾驶功能故障"

状态机的进入退出条件

T1: Initialization → Standby, ALL

序号 信号逻辑 工程含义
a UCM_SensorFault = Normal 12个超声波探头全部自检正常
b EPS_ErrorFlag = Normal && EPS_APStatus = Available EPS无故障且支持自动泊车控制
c ESP_ESPFault = Normal && ESP_APAvailable = Available ESP系统正常且AP功能可用
d ADAS_ECUFaultStatus = Normal ADAS控制器自身无永久性故障
e APA_EntryFlag = TRUE 车辆确实通过APA功能泊入当前车位
f MinClearance_Front ≥ 50cm && MinClearance_Rear ≥ 50cm 前后障碍物距离满足至少0.5m的摆动空间
g ESP_VehicleStandstill = TRUE 车辆处于完全静止状态（车速<0.5km/h)
h BCM_AllDoorsClosed = TRUE && BCM_HoodAjarStatus = Closed && PLG_RearLatchPosition = Locked 四门+前舱盖+后尾门全部关闭锁止
i ADAS_APDriverInCarStatus = DriverInCar 驾驶员座椅压力传感器检测到驾驶员在位
j ACU_DriverBuckleStatus = Buckled 主驾安全带已系紧

Standby → Initialization, ANY

序号 信号逻辑 工程含义
a APA_EntryFlag = FALSE 车辆断电后重新上电，APA记忆丢失
b MinClearance_Front < 30cm || MinClearance_Rear < 30cm 泊入后障碍物移动导致空间不足
c ESP_VehicleStandstill = FALSE 车辆意外移动（溜车)
d HU_APOControl = Disable 驾驶员在HMI主动取消
e ADAS_APDriverInCarStatus = DriverNotInCar 驾驶员离开座位
f BCM_AnyDoorAjar = TRUE 门/盖打开信号
g ACU_DriverBuckleStatus = Unbuckled 安全带解开

T3: Standby → Running, ALL

序号 信号逻辑 工程含义
a HU_APOControl = Right || HU_APOControl = Left 驾驶员选择驶出方向
b IB_BrakePedalStatus = NotPressed 制动踏板处于松开状态，确保控制权无缝交接

T4: Standby → Prerunning, ALL

序号 信号逻辑 工程含义
a HU_APOControl = Right || HU_APOControl = Left 驾驶员选择驶出方向
b IB_BrakePedalStatus = Pressed 制动踏板被踩下，需等待释放

T5: Prerunning → Running, ALL

序号 信号逻辑 工程含义
a IB_BrakePedalStatus = NotPressed 驾驶员已松开制动踏板
b PressedDuration < 10000ms 在10s超时窗口内释放，防止功能挂起

T6: Running → Pause, ANY

序号 信号逻辑 工程含义
a IB_BrakePedalStatus = Pressed && PressedDuration < 6000ms 驾驶员轻踩制动踏板(<6s)进入暂停

T7: Pause → Running, ALL

序号 信号逻辑 工程含义
a HU_SuspendFollowupAction = Resume 驾驶员点击"继续"按钮
b IB_BrakePedalStatus = NotPressed 制动踏板已完全松开

T8: Running/Pause/Prerunning → Quit, ANY

序号 信号逻辑 工程含义
a IB_BrakePedalStatus = Pressed && PressedDuration ≥ 6000ms 持续踩制动>6s，判定为驾驶员意图退出
b EPS_EPSTouqbar > 3.0Nm && Duration > 200ms 驾驶员转向干预扭矩超过阈值
c VCU_CH_AccelPedalPosition > 5% 驾驶员油门介入，V1.6版本由3%调整为5%以减少误触发
d VCU_CH_GearControlFeedback = ControlledbyDriver 驾驶员主动换挡
e ADAS_APDriverInCarStatus = DriverNotInCar 驾驶员离座
f BCM_AnyDoorAjar = TRUE 门盖打开
g HU_SuspendFollowupAction = Quit 暂停状态下点击取消
h ESP_RoadGradient > 10° 或 ObstacleHeight > 8cm 坡度过大或障碍物不可跨越
i ACU_DriverBuckleStatus = Unbuckled 安全带解开

T9: Pause → Prerunning, ALL

序号 信号逻辑 工程含义
a HU_SuspendFollowupAction = Resume 驾驶员点击继续
b IB_BrakePedalStatus = Pressed 制动踏板仍被踩下，需重新等待释放

T10: Running → Finish, ALL

序号 信号逻辑 工程含义
a VehiclePose.x ∈ [-0.5m, 0.5m] && VehiclePose.y > 5.0m 车辆中轴线与车位中心偏差<0.5m，且纵向驶出距离>5m
b EPS_SteeringAngle < 5° 方向盘回正，确保车辆直线行驶

T11: AnyState → Failure, ANY

序号 信号逻辑 工程含义
a EPS_ErrorFlag = HeavyFailure EPS严重故障
b ESP_ESPFault = Failure ESP系统故障
c UCM_SensorFault ≥ 6 超过6个超声波探头故障
d ADAS_ECUFaultStatus = Failure ADAS控制器永久性故障
